# Kannix

A web-based kanban board where each ticket has an attached tmux terminal session, accessible via the browser. Configurable lifecycle hooks run commands when tickets are created, moved, or deleted.

## Architecture

- **Backend**: Python + FastAPI (async, WebSocket support)
- **Frontend**: HTMX + server-side Jinja2 templates + xterm.js for terminals
- **Terminal**: xterm.js in browser ↔ WebSocket ↔ FastAPI ↔ tmux (shared server, sessions named by ticket ID)
- **Storage**: JSON file for kanban state (tickets, columns, users)
- **Config**: JSON config file for columns, hooks, and server settings
- **Auth**: API tokens + simple login (admin creates users, token-based sessions)
- **Packaging**: Nix flake + NixOS module

## Development Practices — TDD + Strict Tooling

- **TDD**: Write failing tests FIRST → implement → green → refactor
- **mypy --strict**: All code fully typed from day one, no `Any` escapes
- **ruff**: Linting + formatting enforced, `ruff check` + `ruff format --check`
- **pytest**: Test runner, pytest-asyncio for async, httpx for API tests
- **Gate**: Every step ends with `ruff check && ruff format --check && mypy --strict src/ && pytest` all green
- **Test structure**: `tests/` mirrors `src/kannix/`, e.g. `tests/test_config.py` for `src/kannix/config.py`

## Key Concepts

- **Columns**: Configurable via JSON
- **Tickets**: title, description, assigned user, column, tmux session name
- **Hooks**: Shell commands on create/move/delete with env vars (TICKET_ID, TICKET_TITLE, TICKET_COLUMN, TICKET_PREV_COLUMN, TMUX_SESSION)
- **Terminal**: Each ticket gets a tmux session, WebSocket streams pty I/O
